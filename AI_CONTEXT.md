# AI Context & Development Guidelines

This file serves as a context provider and guidelines reference for AI agents (and human developers) performing edits or implementing new features in the **GrantTrail** codebase.

---

## 🚀 1. Bootstrapping & Local Environment

Always run commands from the repository root:
* **Setup**: `npm run setup` (installs dependencies in `frontend/` and provisions `frontend/.env.local`).
* **Database**: `npm run db:start` (starts local Supabase containers and applies migrations + seed data).
* **Development Server**: `npm run dev` (starts the React Vite development server).
* **Production Database Deployment**: `npm run db:deploy` (runs the interactive CLI script to link, migrate, and bootstrap a remote production database).

---

## ⛁ 2. Database & Migrations Workflow

### Schema Changes (Local-First)
* **Never** modify database schemas directly on production or cloud environments.
* To make database changes:
  1. Modify your local database via the Supabase Studio dashboard at `http://127.0.0.1:54323` or raw SQL scripts.
  2. Generate a new migration file:
     ```bash
     supabase db diff -f your_migration_name
     ```
  3. Commit the new file in `supabase/migrations/` to Git.

### ⚠️ Synchronizing `seed.sql`
> [!IMPORTANT]
> Whenever you modify table structures, rename columns, add constraints, or alter data types:
> * **You MUST update [supabase/seed.sql](file:///home/ryan/Documents/grant-trail/supabase/seed.sql) to align with the new schema.**
> * If you skip this, `supabase db reset` (which developers run to sync changes) will fail because the mock data insert statements in `seed.sql` will mismatch the new schema.
> * Always test by running `npm run db:reset` to verify a clean reset and seed execution.

### Trigger-Based Side Effects
Do not duplicate logic in the frontend for behaviors managed by database triggers:
* **Status History**: Do not write queries inserting into `grant_status_history`. The `trg_grant_status_tracking` trigger automatically logs status changes when `grant_record.status` is updated.
* **Aggregations & Spending**: Do not manually compute or update `grant_record.total_spent`, `grant_record.remaining_balance`, or `budget_items.amount_spent` on the frontend. Triggers automatically recalculate these values when `expenses` or `budget_items` are inserted, updated, or deleted.
* **Notifications**: In-app notifications in the `notifications` table are automatically populated via database triggers on status updates, comments, and new submissions.

## 🔑 3. Authentication & User Identifiers

The application maintains a strict separation between **Auth Users** and **Profile Records**. Be highly careful with variable naming and lookups:

* **Supabase Auth UUID** (`session.user.id`):
  * A standard string UUID generated by Supabase GoTrue.
  * **Only** use this UUID when querying the `users` table to locate a profile.
* **Database User ID** (`session.userRecord.id`):
  * An auto-incrementing integer (Primary Key of the `users` table).
  * **This integer is the foreign key** used in all other application tables (e.g. `grant_record.user_id`, `expenses.user_id`, `audit_log.user_id`).
  * **Never** insert the string UUID into columns expecting the integer user ID.

## 🛡️ 4. Row Level Security (RLS) & Query Safety

* **Silent Failures**: Supabase RLS policies filter rows at the database level. If a user attempts to access a resource they do not have permissions for, the query returns **`data: null` and `error: null`** rather than throwing an exception.
* **Safe Dereferencing**: Always perform null checks on returned data before referencing properties (e.g. `if (!data) { setError('Not found'); return; }`).
* **`.single()`**: Use `.single()` when querying a row that must be unique. Keep in mind that `.single()` returns an error if no rows are found, so handle `error` objects gracefully.

## 📦 5. File Storage & Compensating Transactions

* **Signed URLs**: All documents and receipts in Supabase Storage buckets are private. Access them by generating short-lived signed URLs (e.g. 60-second expiration) using `supabase.storage.from(...).createSignedUrl(...)`. Do not expose raw storage links in UI hrefs.
* **Orphan Prevention**: React/Supabase client does not support multi-service transactional rollbacks. If a file is uploaded to storage, but the corresponding database record fails to insert:
  * **You MUST write a compensating transaction** in the `catch` block to delete the orphaned file from Storage.
  * *Example reference pattern:* See [GrantAttachments.js](file:///home/ryan/Documents/grant-trail/frontend/src/components/GrantAttachments.js) or [AddExpenseModal.js](file:///home/ryan/Documents/grant-trail/frontend/src/components/AddExpenseModal.js).

## 🎨 6. Frontend CSS & Styling Standards

* **Design Tokens**: All styling properties (colors, border radii, shadows, typography, margins) are declared as CSS custom properties in [variables.css](file:///home/ryan/Documents/grant-trail/frontend/src/styles/variables.css).
* **Hardcoding Hex Values is Forbidden**: Do not write raw hex/rgb values or hardcoded spacing pixels in component-specific stylesheet files.
  * *Correct*: `color: var(--color-primary);`
  * *Incorrect*: `color: #063F1E;`
* **Tailwind**: Do not introduce Tailwind utility classes unless explicitly requested by the user.

## ⚡ 7. React Patterns & Optimization

* **`useCallback` + `useEffect`**: When fetching data from an asynchronous method that also needs to be triggered manually (e.g. on form submit or refresh button), wrap the method in `useCallback` and pass it to `useEffect` dependencies. This prevents rendering loops.
* **O(1) Set Lookups**: When checking for membership or status flags in lists, transform raw arrays into a `Set` (e.g. `new Set(pendingIds)`). Avoid calling `.find()` or `.includes()` on arrays inside map loops.
* **Batching Queries (`.in()`)**: Avoid N+1 database queries. If you need details for related records, fetch parent records first, collect their IDs, and make a single database call using `.in('parent_id', ids)`.
